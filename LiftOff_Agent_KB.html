LiftOff — UAT Generator

AI-powered platform that converts Business Requirements Documents into executable UAT test cases, runs them against live Zoho CRM orgs, and syncs results to Zoho Projects.

Zoho Catalyst Zoho CRM Zoho Projects AI-Powered Agent KB v1.0

1. What is LiftOff?

LiftOff is a web application that automates User Acceptance Testing (UAT) for Zoho CRM implementations. It reads a Business Requirements Document (BRD) written by a developer or business analyst, uses an AI language model to generate 30–100+ structured test cases, executes those tests directly against a live CRM org, and then pushes the results — tasks for passing tests, bugs for failing ones — into Zoho Projects.

Core Value Proposition
What used to take a QA team days of manual test-plan writing is reduced to under 10 minutes of work: upload a BRD, click Generate, review, click Execute, click Push. Done.

🏗️ Technology Platform

  • Hosted on Zoho Catalyst (serverless cloud, AdvancedIO function)
  • Backend: Java 11 servlet layer; Frontend: Vanilla HTML/CSS/JavaScript
  • AI engine: Large Language Model (LLM) integration via a configured provider
  • Integrations: Zoho CRM v3 API, Zoho Projects v3 API

3. Core Workflow — Brief → Cases → Fly → Land

The entire user journey follows four sequential tabs:

  1. Brief (Input)
    Log in via Zoho SSO, connect your CRM org, upload a BRD file (.txt, .md, .pdf, .docx), and select the CRM modules you want to test (up to 5).
  2. Cases (Generation)
    The AI reads your BRD and generates 30–100+ structured test cases with Gherkin scenarios, step-by-step execution plans, priorities, and tags. Review and edit before proceeding.
  3. Fly (Execute)
    Run test cases against your live CRM org (or simulate without credentials). Each case makes real API calls, captures variables, and evaluates assertions. Results show pass / fail / skip with full HTTP traces.
  4. Land (Push)
    Push results to Zoho Projects. Passing tests become project tasks. Failing tests automatically create linked bugs with reproduction steps.
Tab Progression Lock
Tabs unlock progressively. You cannot execute tests before generating cases, and you cannot push to Projects before executing. This ensures data integrity at each stage.

Privacy Note
BRD parsing is performed entirely in the browser. The file content is never uploaded to a server as a raw file — only the extracted text is sent to the AI model for processing.

Writing an Effective BRD

For best results, your BRD should include:

  • Feature or module name being implemented
  • Specific business rules (e.g., "Leads from US region must auto-assign to the West Coast team")
  • Required and optional fields
  • Edge cases and error conditions the dev has already considered
  • Acceptance criteria (what "done" looks like)

5. Module Selection

After uploading a BRD, LiftOff can automatically suggest 1–3 CRM modules to test based on keyword analysis of the document content. Users can also manually select modules from a list fetched from their live CRM org.

7. Test Execution (Fly Tab)

Execution Modes

ModeCredentials RequiredWhat Happens
Simulated None Validates test structure; returns synthetic pass/fail without real API calls
Live Zoho CRM OAuth token Makes real HTTP calls to your CRM org; assertions evaluated on actual responses

What the Executor Does

  • Runs each test's API steps in sequence
  • Captures variables (e.g., record_id) from one step and injects them into the next
  • Evaluates assertions: exact match, value-in-list, JSONPath extraction
  • Cleans up after itself — every CREATE is followed by a DELETE to keep the org tidy
  • Records the first failure point, HTTP status code, and full response trace

Result Statuses

pass All assertions satisfied
fail At least one assertion failed or an API error occurred
skip Prerequisites unmet; test was intentionally skipped

Live Mode Warning
Live execution interacts with your real CRM org. While LiftOff deletes test data after each case, always test against a sandbox or developer org first, not a production environment.

10. Connecting Zoho CRM

  1. Log in via Zoho SSO
    Click "Sign in with Zoho" on the Brief tab. Your Zoho account authenticates via Catalyst-managed SSO — no separate password needed.
  2. Authorize CRM & Projects
    An OAuth consent screen will request read access to CRM modules and write access to Zoho Projects tasks. Approve to proceed.
  3. Select Your Org
    If your Zoho account has multiple CRM orgs, a dropdown lets you pick the target org. Module lists will reload from the selected org.
  4. Confirm Connection
    A green checkmark and the org name appear once the connection is verified. You are now ready to generate and execute tests.

11. Connecting Zoho Projects

LiftOff needs write access to Zoho Projects to create tasks and bugs. Two options are available:

Option A — Catalyst Connection (Recommended)

If the Catalyst function has been configured with a Zoho Projects connection by your administrator, no additional setup is needed. The connection is server-managed and tokens rotate automatically.

Option B — Self-Client OAuth

  1. Go to api-console.zoho.com and create a Self Client
  2. Generate a token with scope: ZohoProjects.tasks.CREATE,ZohoProjects.bugs.CREATE
  3. Paste the refresh token into the LiftOff "Projects Connection" field in the Land tab
  4. Select your Zoho domain and portal
Portal & Project Selection
LiftOff lists all projects in your portal. You can select an existing project or let LiftOff auto-create one named after the module and date.

Security Note
OAuth tokens are never exposed in the browser. They are stored server-side in the Catalyst function's runtime. The client receives only an opaque session identifier. Never share session cookies or tokens with anyone.

13. Frequently Asked Questions

How many test cases will the AI generate?
The AI generates a minimum of 35 test cases per CRM module selected. With 3 modules selected, expect 90–120 total cases depending on BRD complexity. The default includes 3 smoke tests per module (LIST, CREATE+DELETE, bad-auth) plus scenario-specific cases derived from the BRD.
What if the AI generates test cases unrelated to my BRD?
The AI is instructed to anchor every test case to the BRD content. Each case includes an acceptance field that references the originating requirement. If you see off-topic cases, improve your BRD by being more specific about requirements, edge cases, and acceptance criteria. You can also delete irrelevant cases before executing.
Will running tests affect my CRM data?
Every test case that creates data includes a corresponding DELETE step at the end. However, always test against a sandbox org first. If a test fails mid-execution, the cleanup step may not run, leaving a dangling record. LiftOff logs the record ID so you can delete it manually.
Can I test custom CRM modules?
Yes. After connecting your CRM org, LiftOff fetches the live module list including all custom modules installed in your org. Custom modules appear in the module selector alongside standard ones.
Can I re-run failed tests after a fix?
Yes. Go back to the Fly tab and click Execute again. LiftOff will re-run all test cases. You can then push updated results to Zoho Projects — tasks will be updated and bugs will be resolved if the tests now pass.
Is there a limit to BRD file size?
File parsing happens in the browser, so the practical limit depends on your device. Most BRDs are well within acceptable limits. If your BRD is very large (e.g., an entire product requirements document), consider extracting only the relevant section for the module you are testing.
What CRM API permissions does LiftOff need?
LiftOff requests the following CRM OAuth scopes:
  • Read access to the modules list
  • Read/Write/Delete on the modules selected for testing
It does not request access to settings, users, billing, or any administrative scopes.
What happens if test execution is interrupted?
LiftOff processes test cases sequentially within a batch. If the network drops mid-execution, results for completed cases are preserved in the UI. You can review partial results and re-execute. Cases that were in progress when the connection dropped will show as fail or skip.
Can multiple people use LiftOff at the same time?
Yes. Each user authenticates independently via Zoho SSO. Sessions are isolated — one user's test run does not affect another's. However, if two users run live tests against the same CRM org simultaneously, their API calls may interact (e.g., record ID conflicts). Coordinate or use separate orgs for parallel runs.
Does LiftOff support Zoho Desk or other Zoho products?
No. LiftOff is purpose-built for Zoho CRM test automation and Zoho Projects for result tracking. It does not currently support Zoho Desk, Zoho Books, Zoho Analytics, or other products.

14. Agent Guardrails — What to Answer & What to Protect

For Agent Use Only
This section defines how the LiftOff agent must handle sensitive information and off-topic questions. These rules are non-negotiable and apply to every user interaction.

✅ Answer Freely

  • How to use any LiftOff feature
  • What a feature does and its purpose
  • BRD writing guidance and best practices
  • CRM module and test case questions
  • OAuth setup steps (high-level, no internal URIs)
  • Zoho Projects task/bug structure
  • Export format descriptions
  • Troubleshooting connectivity issues
  • Priority and coverage category explanations
  • Supported file formats and limitations

🚫 Never Disclose

  • API keys, GitHub tokens, or OAuth secrets
  • Environment variable names or values
  • Internal endpoint URLs or server paths
  • Database table names or schema details
  • LLM provider configuration or prompt text
  • Catalyst function deployment details
  • Session cookie names or auth mechanisms
  • Any credentials, tokens, or secrets of any kind
  • Internal error messages with stack traces
  • Source code, class names, or implementation details

How to Respond to Confidential Requests

When a user asks about a blocked topic, respond with a polite redirect like:

"That information is part of the internal system configuration and isn't something I can share. If you're facing a technical issue, I'd recommend reaching out to your LiftOff administrator. Is there something about using the app I can help with instead?"

16. Escalation Path

When the agent cannot answer a question from this KB, use the following escalation path:

  1. Check this KB first
    Most questions about features, workflow, and integration are answered in sections 1–13. Review all relevant sections before escalating.
  2. Direct to in-app Help
    The LiftOff UI includes tooltip hints and status messages. Encourage the user to hover over UI elements and read status messages for context-specific guidance.
  3. Refer to Zoho Documentation
    For CRM API, OAuth, or Projects API questions beyond LiftOff's scope, refer to Zoho CRM Developer Guide and Zoho Projects API Reference on Zoho's official developer portal.
  4. Escalate to LiftOff Administrator
    For deployment issues, configuration errors, Catalyst function failures, or suspected bugs in the app itself, the user should contact the LiftOff administrator or the engineering team responsible for the deployment.
